# AI版 / 人間版 対比例

実際の文章でAI臭がどう出て、どう直すか。各例に、関連する構造パターン(`structures.md`)へのポインタを付けた。「人間版」は修正の発想を盗むためのサンプル。正解として丸暗記しない。

## 例1: 技術ブログの序文

→ structures.md: 序文の決めつけ開幕、一文圧縮

**AI版**
> 「Markdownで書けばAIが読みやすいはず」──そう思って CLAUDE.md や RAG のソースドキュメントを Markdown で整備してきた人は多いはずだ。しかし実際にプロンプトエンジニアリングを突き詰めていくと、HTMLやXMLのほうがAIの解釈精度が上がるという経験則に直面することがある。

含まれるAI臭:
- 序文での「」+ 仮想内心
- 全角ダッシュ ──
- 「多いはずだ」の決めつけ
- 「経験則に直面することがある」のヘッジ
- フック→反転構造

**人間版**
> 最近、Markdownで書いておけばAIに伝わるだろうと思って書いた指示書が、思ったほど精度高くなかった、ということがあった。試しに同じ内容をXMLで書き直したら、明らかに反応が良くなって、これってどういう仕組みなんだろうと気になっている。

修正点:
- 「自分はこう思っていたが、こうだった」と一人称で書き出す
- ダッシュなし
- 「らしい」「気がする」型のヘッジは残す
- フック→反転を避け、自分の経験から自然に問題提起

## 例2: 体験記の一節(false agency + 壮大化)

→ structures.md: 主体を消す(false agency、体験の壮大化)、中間温度の欠如

**AI版**
> JALの機内で出されたお弁当が普通に美味しすぎた。これ、ただの機内食じゃなくて、しっかり「設計」されてるんですよね。何時間か後に着くカンファレンスでの集中力まで考えて作られている。参加者体験という観点で見ると、イベント設計としてかなりレベルが高いんだなって感じました。

含まれるAI臭:
- 「美味しすぎた」(中間温度なし)
- 「普通に」(根拠なし副詞)
- 「設計」のカギ括弧囲み + 体験への適用 = false agency
- 体験→抽象化のテンプレ(お弁当 → イベント設計論)
- 「○○なんだなって感じました」型の締め
- 「観点で見ると」の翻訳調
- 主体の隠蔽(誰が設計したのか書かない)

**人間版**
> JALの機内で出されたお弁当が思ったより美味かった。主催者が選んだのか航空会社のメニューなのか分からないけど、機内食であのレベル出されると、なんかちゃんとしてるなと思う。

修正点:
- 「思ったより美味かった」(中間温度)
- 「普通に」削除
- 「設計」を「主催者が選んだのか航空会社のメニューなのか」に展開して主体を明示
- 抽象化を保留(「ちゃんとしてる」止まり)
- 「分からないけど」の保留を残す

## 例3: 解説記事の事実説明(一文圧縮)

→ structures.md: 一文圧縮(Wikipedia調)

**AI版**
> JavaScriptは1995年にBrendan Eichが10日間で設計した。この言語は瞬く間に普及した──しかし型安全性は設計目標に含まれていなかった。これがTypeScriptが生まれた背景である。

含まれるAI臭:
- 一文圧縮(年代・人名・期間・評価を1文に)
- 断定型(「○○した」「○○である」)
- 全角ダッシュ ──
- 翻訳調(「設計目標に含まれていなかった」)
- Wikipedia/辞書のような口調

**人間版**
> JavaScriptは、もともと1995年にBrendan Eichが作った言語です。およそ10日くらいで設計されたそうです。そこからまたたく間に普及し、今のWebの標準になりましたが、開発当時は型安全みたいなことはあんまり考えられなかったようで、それがあとからTypeScriptが生まれた背景になっています。

修正点:
- 伝聞口調(「そうです」「ようで」)を混ぜる
- 一文を分割
- ダッシュなし
- 「もともと」「あとから」のような曖昧な接続詞を入れる

調べた事実には伝聞表現を入れるのが人間の自然な書き方。直接体験したことや知っていることは断定でよい。

## 例4: false agency の修正(技術記事)

→ structures.md: 主体を消す(false agency)

**AI版**
> Pythonコミュニティでは、長年Type Hintsの議論が続いている。一方でTypeScriptの成功事例から、静的型付けの価値が再認識されるようになった。

含まれるfalse agency:
- 「議論が続いている」(誰が議論しているのか書かない)
- 「価値が再認識されるようになった」(誰が再認識したのか書かない)

**人間版**
> Python界隈の人たちは、何年もType Hintsをどうするかで議論している。TypeScriptがうまくいったのを見て、静的型付けって意外と良いのかも、と思う人が増えてきたっぽい。

修正点:
- 「人たち」を主語に立てる
- 「議論している」(現在形にすると主体が出る)
- 「思う人が増えてきたっぽい」(誰が思っているのか明示 + 伝聞)

## 例5: 中間温度の混在(レビュー記事)

→ structures.md: 中間温度の欠如

**AI版**
> 今回試したフレームワークは本当に素晴らしかった。開発体験が劇的に向上し、生産性が爆上がりした。これは間違いなくゲームチェンジャーだ。

**人間版**
> 今回試したフレームワーク、悪くなかった。一部は確かに楽になったけど、思ったほど劇的ではない感じ。慣れもあると思うけど、もう少し使ってみないと分からない部分がある。

修正点:
- 「素晴らしかった」「劇的に」「爆上がり」「ゲームチェンジャー」を全部下げる
- 「悪くなかった」「思ったほど劇的ではない」「分からない部分がある」を入れる
- 評価を保留する

## 例6: ムラの欠如(ノウハウ記事)

→ structures.md: ムラの欠如、3項目並列の罠、段落の均一な閉じ方

**AI版**(全段落が同じ長さ、同じ着地)
> 第一に、責任の所在を明確にすることが重要です。これにより、問題発生時の対応速度が上がります。
>
> 第二に、定期的なレビューを実施することが必要です。これにより、品質を継続的に改善できます。
>
> 第三に、ナレッジ共有の仕組みを作ることが効果的です。これにより、組織全体の学習速度が向上します。

**人間版**(段落の長さ・温度・着地がバラバラ)
> 一番大事なのは、誰がやるか決めること。これ決めとかないと、問題起きたときに全員が様子見になる。
>
> あとは、たまにレビューする会を入れる。月1で十分。やらないと感覚がズレてくる。
>
> ナレッジ共有はまあ、難しい。Notion整備するとか色々やったけど、結局のところ書きたい人が書きたいときに書く以上のものはなかなか機能しない気がする。

修正点:
- 段落の長さをバラけさせる
- 並列構造(第一に、第二に、第三に)を崩す
- 最後の段落で結論を保留(「気がする」)
- 「これにより〜が向上します」の定型を避ける

## 例7: 命題型H2の修正

→ structures.md: 命題型H2見出し

**AI版**
```
## Pinecone Nexusもまた同じ方向を指している

実は、Pinecone Nexusというサービスもまた、これまで見てきた他のベクトル検索エンジンと同じ方向性を持っている。具体的には...
```

含まれるAI臭:
- 命題型H2(主張を見出しに)
- H2の言い換えで本文を始める(結論先取り)

**人間版**
```
## Pinecone Nexus を試した

実際に Pinecone Nexus を3週間くらい触ってみた。最初に気になったのは、検索のレイテンシが想像より低かったこと。具体的には...
```

修正点:
- 見出しを名詞句にする
- 本文を具体例から始める
- 抽象主張を避ける
- 自分の経験(3週間触った)を主語に

## 例8: 序文の決めつけ開幕の修正

→ structures.md: 序文の決めつけ開幕(フック→反転)

**AI版**
> Claudeを使えば誰でも質の高い文章が書ける──そう信じて活用してきたライターは多いはずだ。しかし、いざクライアントに納品すると「AI臭い」と返ってくる。なぜか。

**人間版**
> 自分はClaudeで文章書くようになって2年くらい経つけど、最近よく「AI臭い」って指摘されることが増えてきた。直そうとはしてるんだけど、どこが臭いのかちゃんと言語化できてなかったので、調べてみた。

修正点:
- 「多いはずだ」の決めつけを「自分は」に変える

- ダッシュ削除
- 「なぜか」の煽りを「調べてみた」に変える(自然な動機付け)

## 例9: 二項対比の修正

→ structures.md: 二項対比(Not X. It's Y.)

**AI版**
> これは単なる効率化ツールではない。これは思考のパートナーだ。
> AIは置き換えるためのものではなく、拡張するためのものだ。
> 重要なのは速度ではなく、深さである。

含まれるAI臭:
- 二項対比の連続
- 「単なる○○ではなく」の構文
- 「○○ではなく、△△だ」の telegraphed reversal

**人間版**
> AIで文章書いていると、自分の考えを言語化する助けになる。何かを置き換えるというより、考えを広げる感じ。早く書けるかどうかより、ちゃんと考えられるかどうかが大事だと思う。

修正点:
- 二項対比を直接的な肯定文に変える
- 「思考のパートナー」のような大袈裟な表現を「考えを広げる感じ」に
- 「重要なのは○○ではなく」を「○○が大事だと思う」に

## 例10: 否定的列挙 + 劇的断片化の修正

→ structures.md: 否定的列挙、劇的な断片化

**AI版**
> これは小手先のテクニックではない。一時的な流行でもない。働き方そのものの変化だ。
> 速く。深く。そして静かに。それが、これからのスタンダードになる。

含まれるAI臭:
- 否定的列挙(「Aではない、Bでもない、Cだ」)
- 劇的断片化(体言止めの連発で間を演出)
- 「それが、〜になる」の予言調

**人間版**
> これはたぶん働き方そのものが変わる話だと思う。派手さはないけど、気づいたら当たり前になってるタイプの変化な気がする。

修正点:
- 否定列挙をやめ、「働き方が変わる」を直接書く
- 断片化した体言止めを普通の文に戻す
- 予言調を「気がする」の保留に下げる

## 例11: 体験記での教訓の抑制

→ structures.md: 体験の壮大化、主体を消す、遠くから語る話者

**AI版**
> ホテルの大浴場で湯船に浸かりながら、ふと気付いた。ここに来るまでの忙しなさが、嘘のように溶けていく。これこそが、現代社会で失われつつある「余白」というものなのかもしれない。我々は、もっとこういう時間を意識的に作るべきなのだろう。

含まれるAI臭:
- 体験の壮大化(風呂 → 「余白」「現代社会」)

- 「○○なのかもしれない」「○○すべきなのだろう」のメタな教訓化
- 「我々は」の上から目線

**人間版**
> ホテルの大浴場、思ったよりずっと良かった。ここまでの移動疲れがちょっと取れた気がする。月一くらいで温泉行ければなあ、と思った。

修正点:
- 教訓を抜く
- 「我々は」を「自分」(あるいは省略)に
- 「現代社会」「余白」などの抽象語を削除
- 普通の感想で終わらせる

## 例14: disclaimer儀式の除去

→ structures.md: リチュアル化したdisclaimer

**AI版**
> 私のチームではこの方法でリリース頻度が上がった。ただし、これはあくまで個人の経験に基づく見解であり、定量的に検証したものではない。組織の規模や文化によって結果は異なる可能性があるため、参考程度に捉えていただきたい。

含まれるAI臭:
- 結論のあとに保険を3連発
- 「あくまで個人の経験」「可能性がある」「参考程度に」のヘッジ重ね
- せっかくの主張(リリース頻度が上がった)を自分で打ち消している

**人間版**
> ウチのチームではこの方法でリリース頻度が上がった。10人くらいの規模での話なので、もっと大きいチームだとどうかは分からない。

修正点:
- 保険を1つに減らす(「10人くらいの規模での話」)
- 保険を具体的な適用範囲として書く(抽象的なヘッジを避ける)
- 主張は打ち消さずに残す

## 例15: 必殺技造語(キーボード比較を膨らませる)

→ structures.md: 体験の壮大化、必殺技造語による壮大化

**AI版**
> エンジニアとして三年、二台のキーボードと過ごす日々。HHKBの「禁欲的」な打鍵感と、REALFORCEの「優美な」反発。指先から伝わる感触の違いを毎日確かめ続けた末に辿り着いた、ある「真実」。スペックや価格という「虚飾」を剥ぎ取った時、僕の指先が感知した違和感の正体とは——。

含まれるAI臭:
- 必殺技造語の連発(「禁欲的」「優美」「真実」「虚飾」)
- 全角ダッシュ ──
- 形容詞・心情への「」乱用
- 「指先が感知」のfalse agency
- 結論を先送りする煽り(「正体とは——」)
- 3年使ったという個人の話を「真実」「虚飾」レベルに膨らます

**人間版**
> HHKBとREALFORCEを3年使ってきたけど、結局HHKBのほうが好きだった。REALFORCEは打鍵感が柔らかすぎる気がする。

修正点:
- 「真実」「虚飾」「禁欲的」「優美」を全部削る
- 「指先が感知した違和感」を「打鍵感が柔らかすぎる気がする」に
- ダッシュと煽りを削除
- 「3年使った末の真理」ではなく「3年使って好きだった」に着地

## 例17: 全方位肯定の批判(ツール比較記事)

→ structures.md: 無害さにチューニングされている - 全方位肯定

**AI版**
> Notion、Obsidian、Logseqそれぞれに独自の強みがあり、用途やワークフローによって最適な選択は異なります。Notionはチーム共有に優れ、Obsidianはローカル管理とリンク機能が強力で、Logseqはアウトライナーとしての完成度が高いと言えるでしょう。

含まれるAI臭:

- 「それぞれに独自の強み」のテンプレ
- 3項目並列
- 「と言えるでしょう」の弱いヘッジ
