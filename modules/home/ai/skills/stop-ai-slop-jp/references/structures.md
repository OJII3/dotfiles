# 避けるべき構造パターン

ここに挙げるのは、語彙より深い場所でAI臭を出している構造のクセ。フレーズだけ直しても、構造が残れば臭いは消えない。`phrases.md` が「どの単語を避けるか」なら、こちらは「どの形を避けるか」を扱う。

それぞれが生む「偽物」で6系統に束ねた。

1. [主体を消す](#1-主体を消す) — 誰がやったかを書かない
2. [ドラマを偽造する](#2-ドラマを偽造する) — 起伏のない話に起伏を演出する
3. [英語ブログの骨格を直輸入する](#3-英語ブログの骨格を直輸入する) — 日本語に合わない型を持ち込む
4. [リズムを均す](#4-リズムを均す) — 揺らぎを消して機械的にする
5. [立場から逃げる](#5-立場から逃げる) — 何も引き受けない
6. [無害さにチューニングされている](#6-無害さにチューニングされている) — RLHFの安全弁が漏れ出している

---

## 1. 主体を消す

AI臭の構造的な核心。AIは**誰がやったのかを名指しすることを避け**、無生物に人間の動詞を当てる。人間が書く文章では必ず誰かが何かをしている。AIが書く文章では、状況や概念が勝手に動く。

### 1-1. False Agency(モノに人間の動作をさせる)

無生物が能動的な動詞を取っていたら、必ず「誰が」を考え、それを主語に立て直す。

| AI版(モノが主語) | 人間版(人が主語) |
|---|---|
| データが示している | データを見るとこうなってる |
| 文化が醸成される | 経営陣が文化を作った |
| 課題が浮き彫りになる | やってみて課題に気づいた |
| 思想が結実する | 設計者がそう作った |
| 複雑性が増していく | 機能を足しすぎた |
| 自然と最適化されていく | チームで少しずつ直した |
| 市場が報いる | 顧客が買ってくれる |
| 議論が深まる | 話してたら細かい話まで出てきた |
| 解像度が上がる | 分かるようになってきた |
| アーキテクチャが導く | 設計者がそう決めた |
| プロセスが定着する | みんながやるようになった |

具体的な「誰が」が思いつかないなら、その文は具体性が足りない。書き手が誰のことを語っているのか分かっていない可能性が高い。`phrases.md` の「翻訳調動詞」(指している、示している、収斂する)も、この主体隠しの語彙的な現れ。

### 1-2. 体験の壮大化

個人の小さな体験を抽象命題に変換するクセ。体験そのものを主語にして「設計」「装置」を当てるのも、false agencyの一種。お弁当が設計したわけでも、大浴場が装置として機能したわけでもない。誰かがそれを意図して用意した。

| AI版 | 人間版 |
|---|---|
| カフェのコーヒーが、職人の哲学を一杯に凝縮している | 店主がこだわってコーヒーを淹れている |
| 街中の散歩が、都市の動線と建築の対話を浮かび上がらせる | 街を歩いていると、よく考えられた配置だなと感じる |
| 雑談が、組織知の伝達装置として機能していた | みんな雑談で意外な情報を持ち帰ってる |
| 大浴場で副交感神経が優位になり、まさに回復装置だった | 大浴場気持ちよかった、よく寝れた |

体験記では、すべてのエピソードに教訓を付けない。「美味かった」「気持ちよかった」「久しぶりで嬉しかった」で終わるエピソードを残す。教訓を付けるのは1記事に1〜2回まで。

#### 必殺技造語による壮大化

壮大化の語彙的なエンジン。普通の感想を「真理」「結末」「境地」「虚飾」「美学」のような必殺技みたいな大きな名詞で括るクセ。個人の小さな話が、人類の課題か何かみたいに膨らむ。

**AI版**(キーボードを3年使った話を膨らませたパターン)
> エンジニアとして三年、二台のキーボードと過ごす日々。HHKBの「禁欲的」な打鍵感と、REALFORCEの「優美な」反発。指先から伝わる感触の違いを毎日確かめ続けた末に辿り着いた、ある「真実」。スペックや価格という「虚飾」を剥ぎ取った時、僕の指先が感知した違和感の正体とは——。

**人間版**
> HHKBとREALFORCEを3年使ってきたけど、結局HHKBのほうが好きだった。REALFORCEは打鍵感が柔らかすぎる気がする。

「真実」も「虚飾」も「禁欲的」も要らない。

「枯淡の境地」「冷徹な結末」「残酷な真理」「虚飾を剥ぎ取る」「禁欲的な美学」「重厚感の欠如」のような壮大化語彙は `phrases.md` の偏愛語と連動。AIは小説・エッセイ・文芸評論の文体を大量に学習しているので、放っておくとこの手の漢字熟語に寄せたがる。

### 1-3. 一人称の消失と一般論

テーマだけ渡してAIに書かせると、「自分は」「私は」「俺は」が消えて、代わりに「多くの人」「人々」「現代社会」「エンジニアは」のような一般論になる。自分の小さい話を、勝手に世間全体の話にしない。

| AI版 | 人間版 |
|---|---|
| 多くのエンジニアが、TypeScriptの型定義に頭を悩ませている | 自分もTypeScriptの型定義でずっと困ってる |
| プロダクト開発では、フィードバックの優先順位付けが重要である | 去年、同じボタンについて3人から文句を言われて直したら、別の人に前のほうがよかったと言われた |
| ユーザーの声を取り入れることは、継続的な改善につながる | それ以来、フィードバックが来ても3日寝かせるようにしている |

一般論だけの段落は情報量がゼロに近い。具体的な人、具体的なエピソード、具体的な失敗、数値、固有名詞を入れる。入れられないなら、書き手がその話を持っていない。

## 2. ドラマを偽造する

平坦な内容に、構文で無理やり起伏をつけるクセ。英語圏で「telegraphed reversal(予告された反転)」と呼ばれる型の輸入。直接言い切れば済むことを、わざと回り道する。

### 2-1. 二項対比(Not X. It's Y.)

「Xではなく、Yだ」を多用する。直接Yを書けばいい。

| AI版パターン | 修正 |
|---|---|
| 「Aではなく、Bである」 | 「Bだ」と直接書く |
| 「単なるAではなく、Bだ」 | 「Bだ」 |
| 「Aだけじゃなく、Bでもある」 | 「Bだ」 |
| 「実はAではなくBだった」 | 「Bだった」 |
| 「Aに見えるが、Bだ」 | 「Bだ」 |
| 「Aではなくなり、Bになる」 | 「Bになった」 |

対比を残すなら、XとYの違いを具体的に書く。「速いより正確なほうが効く場面だった」のように。「Xではない」と否定で並べる形は避ける。

### 2-2. 否定的列挙(negative listing)

何かを示す前に「何でないか」を並べるクセ。

| AI版 | 修正 |
|---|---|
| 「Aではない。Bでもない。Cだ。」 | 「Cだ」 |
| 「これは戦略ではなく、戦術でもなく、習慣だ」 | 「これは習慣だ」 |
| 「彼は研究者でも、エンジニアでもなく、職人だった」 | 「彼は職人だった」 |

### 2-3. 劇的な断片化(dramatic fragmentation)

短文を連発して深さを演出するクセ。体言止めと句点で「間」を作ろうとする。

| AI版 | 修正 |
|---|---|
| 「シンプル。それだけ。それが本質。」 | 「シンプルだ」 |
| 「速く。正確に。継続的に。」 | 「速く正確に続ける」 |
| 「これがすべてを変える。本当に。すべてを。」 | 「これで状況が変わる」 |

## 3. 英語ブログの骨格を直輸入する

英語SEOブログやMediumの構成を、そのまま日本語に持ち込むクセ。型は機能的だが、日本語の自然な運びと衝突して翻訳調になる。

### 3-1. 命題型H2見出し

見出しに「主張」を置くクセ。テーマの名前で立てるのが本来。`How X solves Y` 構造の輸入。

| AI版 | 人間版 |
|---|---|
| ○○もまた同じ方向を指している | ○○の話 |
| 実践で見えた3つの原則 | やってみての感想 |
| ○○は△△を失った瞬間に死ぬ | △△についての雑感 |
| なぜ我々は○○を選んだのか | ○○を選んだ理由 |
| ○○が私たちに教えてくれること | ○○から学んだこと |

命題型H2は本文に「主張への裏付け」をぶら下げる構造を強制するので、各節が小さなTED talkになる。これがしんどい。

さらに悪化版として、**H2で主張を置き、本文冒頭で同じ主張を言い換えて繰り返す**型がある。

```
## ○○もまた同じ方向を指している

実は、○○というアプローチもまた、これまで見てきた他の手法と同じ方向性を持っている。
```

冒頭の文がH2の言い換えになっている。両方は要らない。**H2は名詞句にして、本文は具体例から入る。**

### 3-2. 序文の決めつけ開幕(フック→反転)

冒頭で強い断定を投げ、すぐ反転させる構造。英語ブログのhook構造の直訳。

**AI版**
> ChatGPTを使えば誰でも記事が書ける。そう信じて活用してきたライターは多いはずだ。しかし——

**人間版**
> 最近、ChatGPTで記事書くより、自分で下書きしたほうが結局早いんじゃないか、と思うようになってきた。きっかけは、

人間は控えめから始めて確信に至る。AIは断言から始めて控えめになる。冒頭で「○○な人は多いはずだ」「誰もが○○を経験したことがあるだろう」のような決めつけがあったら、まず「自分は○○だった」に書き換える。

### 3-3. 一文圧縮(Wikipedia調)

年代・人名・定義・評価を並列で1〜2文に詰めるクセ。

**AI版**
> JavaScriptは1995年にBrendan Eichが10日間で設計した。この言語は瞬く間に普及した——しかし型安全性は設計目標に含まれていなかった。

**人間版**
> JavaScriptは、もともと1995年にBrendan Eichが作った言語です。およそ10日くらいで設計されたそうです。そこからまたたく間に普及して今のWebの標準になりましたが、開発当時は型安全みたいなことはあんまり考えられなかったようです。

人間が事実を書くとき、伝聞口調(「らしい」「だそうです」「ようだ」「と聞いた」「みたい」)が混ざる。自分が直接知っていることはともかく、調べた事実には伝聞表現を入れる。Wikipediaや辞書のような断定型は、人間は基本的に避ける。

### 3-4. メタ的な構造宣言

本文の中で「この記事では〜」「以下〜について説明します」と構造を予告するクセ。見出しで分かることを本文で再宣言しない。

| AI版 | 修正 |
|---|---|
| 「以下の3つの観点から説明します」 | 削除して直接1つ目を書く |
| 「それぞれ順に見ていきましょう」 | 削除 |
| 「ここからが本題です」 | 削除 |
| 「補足ですが、」 | 削除して補足を直接書く |
| 「○○について、改めて整理します」 | 削除 |

H2で「RAGの基本」と書き、H3で「検索フェーズ」と書けば、構造は明らか。本文での再宣言は要らない。

### 3-5. ❌/✅ の bad-then-good 比較

絵文字のチェックとバツで対比する構造。スライドやSEO記事の型。

**AI版**
> ❌ 古いやり方(問題点)
> ✅ 新しいやり方(改善点)

**人間版**
> 古いやり方だとここが面倒で、新しいやり方にしたらここが楽になった。

対比は地の文に展開する。

## 4. リズムを均す

人間の文章には自然な揺らぎがある。AIはそれを均す。均一さは「書き手が疲れていない、迷っていない、関心の濃淡がない」ことを意味し、機械的に感じさせる根本になる。

### 4-1. ムラの欠如

5種類のムラを意識して残す。

- **長さのムラ**: 段落と文の長さをバラバラにする。1行で終わる段落と10行続く段落を混ぜる。
- **密度のムラ**: 興味のある話題には固有名詞・数値・事実を詰め、興味のないところはサラっと流す。
- **トーンのムラ**: 急に冷めた書き方になる、テンションが下がる、面倒くさそうになる。
- **結論のムラ**: ある段落はきれいに着地、別の段落は「で、結局なんだったんだろう」で終わる。
- **詳細度のムラ**: ここだけ詳しい、ここは雑、という偏りを残す。

### 4-2. 段落の均一な閉じ方

全段落を律儀に着地させるクセ。「〜なんだなと感じました」「〜ということが分かりました」「〜の重要性を痛感しました」。

段落の終わり方を意図的にバラけさせる。途中で文を切る、結論を保留する、別の話に逸れる、を混ぜる。

### 4-3. 体験→引用→抽象化のテンプレ反復

体験記で頻出する型。「具体エピソード → 「」で心の声 → 『○○なんだなって感じました』で抽象化」。

これを全セクションで繰り返すと、リズムが完全に均一化する。テンプレが3回連続したら警戒。1回は別の形式にする(教訓なしで終わる、引用なしで体験だけ書く、抽象化を別の段落に独立させる)。

### 4-4. 3項目並列の罠

AIはとにかく3つ並べたがる。「3つのポイント」「3つの観点」「3つの原則」「3つの理由」。

3項目に出会ったら、2つか1つに削れないか考える。本当に3つ必要な状況は意外と少ない。3つ並ぶ時点で、書き手が「3つ揃えるために無理した3つ目」を書いている可能性を疑う。

### 4-5. 文末のバリエーション不足

AIは「である調」と「ですます調」のどちらかを一貫させようとする。一貫させること自体は悪くないが、その中での揺れが消える。人間が書く文章には「だ」「だよね」「らしい」「みたい」「なんすよ」のような口調の小さな揺れが混ざる。

全文を同じ語尾(「〜と思います」「〜である」「〜だ」)で終わらせない。1段落に1〜2回は別の語尾を混ぜる。

### 4-6. 不完全さの不在(誤字脱字・言い淀みのなさ)

人間の文章には、たまに誤字が残り、「まぁ」「なんていうか」「えーと」のような言い淀みが混ざる。AIの出力は完璧で、これが一切ない。

完璧であることそのものがAI臭の指紋になる。書き直す過程で、言い淀みや小さなブレを意図的に残すことを許す。文の途中で考えを変えて切り返す、句点を打ちそびれる、文末を「なんすよ」「だよね」みたいに崩す、をAIに後から消させない。

## 5. 立場から逃げる

AIのデフォルトは「間違えないこと」。そのために評価を振り切るか保険をかけ、結局何も引き受けない。これがAI臭の本体に最も近い系統。

### 5-1. 中間温度の欠如

「悪くない」「ふつう」「まあまあ」「よくわからない」が出てこない。評価がすべて両極端になる。

- AIは全部「最高」「すごい」「すぎる」、否定なら「絶対に避けるべき」「論外」
- 人間は「まあそこそこ」「微妙」「悪くはない」「よく分からんけど好き」を混ぜる

これは「○○すぎる」の連発(お弁当が美味しすぎ、コーヒーが優雅すぎ、マッサージが気持ちよすぎ)としても現れる。本当に強い感情のときだけ強い言葉を使う。普通に良かった程度なら「悪くない」「ふつうに良い」と書く。

### 5-2. リチュアル化したdisclaimer

各セクションの最後に保険文を置く儀式。RLHFで叩き込まれた安全弁の漏れ出し。

| AI版 | 人間版 |
|---|---|
| 「あくまで個人の経験に基づく見解であり、定量的な検証ではない」 | (書かない) |
| 「環境やケースによって結果は異なる可能性がある」 | (書かない、または「ウチではこうだった」とだけ) |
| 「実装はモデルによって異なる」 | (書かない) |

毎節に保険をかけると立場が消える。必要な保留は1記事に1〜2回まで。

### 5-3. 遠くから語る話者(narrator-from-a-distance)

上から目線で社会を論じ、読者を場に置かないクセ。「人々」「誰も」「我々」を「自分」「あなた」に書き換える。

| AI版 | 人間版 |
|---|---|
| 「人々は〜する傾向にある」 | 「自分も含めて、〜してしまう」 |
| 「現代社会において、〜は重要だ」 | 「最近、〜が気になる」 |
| 「誰もが〜を経験したことがあるはずだ」 | 「自分は〜を経験した」 |
| 「組織はしばしば〜する」 | 「自分の会社では〜だった」 |
| 「エンジニアは〜と考えがちだ」 | 「自分は〜と思っていた」 |

## 6. 無害さにチューニングされている

RLHFで「人を傷つけるな」「公平であれ」「両論併記しろ」と訓練された結果、漏れ出てくるお行儀の良さ。書き手の立場の不在(系統5)とは原因が違う(訓練側のクセ)が、読み手には同じく「腰が引けてる」と映る。

系統5はプロンプトと書き手の意識で直せる。系統6はモデル側のクセなので、書き手が手で書き足すしかない箇所が多い。

### 6-1. 全方位肯定(誰にも噛みつかない)

技術記事で「○○というツールも、△△というツールも、それぞれの良さがあります」と書くやつ。両論併記は読み手の判断材料を提供しているように見えて、実は書き手が判断を放棄している。

レビュー・比較・選定の記事では、書き手が「自分はこれを選んだ」「これは選ばなかった」「あれはダメだった」を引き受ける。引き受けないなら、その記事は書かない。

| AI版 | 人間版 |
|---|---|
| 「それぞれに独自の強みがあり、用途による」 | 「自分はこれを使っている。あれはやめた」 |
| 「賛否が分かれるテーマです」 | 「自分はこう思う」 |
| 「メリットもデメリットもあります」 | 「これはいい。これはダメ」 |

### 6-2. 毒・皮肉・自虐・罵倒の不在

人間の文章には毒がある。「あれはマジでクソだった」「自分がアホだった」「あの会社のドキュメントは犯罪レベル」のような振り切れた感情が出る。

AIにはこれが出ない。RLHFで叩かれているので「攻撃的に書いて」と頼んでも限度がある。直すには、書き手側が毒の部分を自分で書き足すしかない。毒・自虐・皮肉が必要な箇所は自分の手で書く。AIに完全自動で書かせたまま読み手に出さない。

ただし「全部毒舌」も別の臭さになる。毒は感情の表明なので、必要な箇所に必要なだけ入れる。
